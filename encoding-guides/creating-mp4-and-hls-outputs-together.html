    Guía de transmuxing

    En este tema se explica la transmuxing, un proceso que le permite codificar sus vídeos en MP4 y luego volver a empaquetarlos como salidas HLS.

    Resumen

    Con Zencoder puedes codificar tus vídeos en MP4 y luego volver a empaquetarlos como salidas HLS, a través de un proceso que llamamos «transmuxing» (en lugar de transcodificación). La transmuxing reempaquetará los vídeos MP4 existentes en los segmentos MPEG TS necesarios para HTTP Live Streaming (HLS), sin tener que volver a codificar los archivos de vídeo. Usando salidas dependientes, puede crear sus archivos H.264 más los archivos segmentados, todo en un solo trabajo, lo que produce un giro más rápido a menor costo; las salidas transmuxadas se cargan a 1/4 del coste de codificación. El trabajo resultante crea 6 salidas: 2 cargadas a la tarifa regular de su cuenta, 3 archivos transmuxados a 1/4 del costo de codificación y 1 lista de reproducción, que es gratuita.

    Para ello, creará un único trabajo con dos conjuntos principales de salidas (más las listas de reproducción). El primer conjunto de salidas serán archivos MP4 H.264/AAC normales con algunos ajustes especiales para permitirles trabajar como archivos fuente para salidas HLS. El segundo conjunto de salidas utilizará esos archivos MP4 como entradas «fuente», transmuxando su contenido a salidas HLS. Dado que estos son salidas dependientes, esperarán hasta que finalice su fuente correspondiente antes de ser programados para procesar.

    La parte clave del proceso es utilizar source, una nueva opción en V2 de la API de Zencoder. Source indica a una salida que utilice el archivo creado por otra salida en el trabajo para su procesamiento, en lugar del archivo de entrada. En esta situación, Zencoder creará archivos H.264 basados en el archivo de entrada a las velocidades de bits solicitadas. A medida que termine cada una de esas salidas, se creará una versión segmentada a partir de la salida.

    Nota: No hay forma de garantizar que la reproducción sea fluida si combina salidas transmux y transcodificación en una lista de reproducción. Es una práctica recomendada que las copias tengan que codificarse de la misma manera.

    Las salidas H.264

    Empezaremos creando 2 archivos H.264, orientados a velocidades de bits altas y bajas.

    {
    "label": "low",
    "format": "mp4",
    "video_bitrate": 200,
    "decoder_bitrate_cap": 300,
    "decoder_buffer_size": 1200,
    "audio_sample_rate": 44100,
    "height": "288",
    "url": "s3://example-bucket/low.mp4",
    "h264_reference_frames": 1,
    "forced_keyframe_rate": "0.1",
    "audio_bitrate": 56,
    "decimate": 2
},
{
    "label": "high",
    "format": "mp4",
    "video_bitrate": 1000,
    "decoder_bitrate_cap": 1500,
    "decoder_buffer_size": 6000,
    "audio_sample_rate": 44100,
    "height": "432",
    "url": "s3://example-bucket/high.mp4",
    "h264_reference_frames": "auto",
    "h264_profile": "main",
    "forced_keyframe_rate": "0.1",
    "audio_bitrate": 56
}

    Los archivos resultantes de estas salidas son capaces de ser reproducidos en una amplia variedad de dispositivos. Cada uno tiene como objetivo una velocidad de bits y una resolución diferentes, por lo que se puede enviar a los usuarios el archivo apropiado. Cada una de ellas también es adecuada para segmentar para HTTP Live Streaming y servir como una transmisión de velocidad de bits adaptable.

    Algunas opciones en la solicitud anterior para tener en cuenta:

    • forced_keyframe_rate a 0.1. Esto obliga al vídeo a tener un fotograma clave cada 10 segundos. Los archivos segmentados tendrán una duración de 10 segundos, por lo que esto garantiza que cada segmento comience con un fotograma clave.
    • decoder_bitrate_cap se establece en 1.5x la velocidad de bits objetivo del archivo. decoder_buffer_size se establece en 3.5x a 5 veces la velocidad de bits objetivo del archivo. Esta configuración ayudará a mantener una velocidad de bits coherente en todo el archivo, de modo que los segmentos segmentados no varían demasiado en tamaño y velocidad de bits.

    Ahora que se han creado los archivos H.264, se pueden agregar salidas adicionales a la solicitud para crear archivos segmentados HTTP Live Streaming desde el origen, sin necesidad de hacer codificación adicional.

    Salidas segmentadas

    Cada una de las salidas de vídeo H.264 y audio anteriores tienen labels en sus opciones API. Estas etiquetas se pueden utilizar junto con la source opción de indicar a Zencoder que utilice el vídeo creado por la salida con la etiqueta dada, en lugar del archivo de entrada. Dado que los archivos H.264 ya están en buenas configuraciones para HLS, no es necesaria ninguna codificación adicional.

    {
    "source": "low",
    "format": "ts",
    "copy_audio": "true",
    "copy_video": "true",
    "url": "s3://example-bucket/hls-low/hls-low.m3u8",
    "label": "hls-low",
    "type": "segmented"
}
{
    "source": "high",
    "format": "ts",
    "copy_audio": "true",
    "copy_video": "true",
    "url": "s3://example-bucket/hls-high/hls-high.m3u8",
    "label": "hls-high",
    "type": "segmented"
}
{
    "streams": [
    {
        "path": "hls-low/hls-low.m3u8",
        "bandwidth": 256
    },
    {
        "path": "hls-high/hls-high.m3u8",
        "bandwidth": 1056
    }
    ],
    "type": "playlist",
    "url": "s3://example-bucket/playlist.m3u8"
}

    Los archivos resultantes de estas salidas son salidas segmentadas, dirigidas a diferentes anchos de banda, además de un archivo de lista de reproducción para que el dispositivo de reproducción conozca las secuencias disponibles.

    Cada salida segmentada incluye source, y especifica una de las salidas enumeradas anteriormente. También incluyen copy_audio y copy_video opciones, que le dicen a Zencoder que use las transmisiones de audio y video del archivo fuente y las vuelva a empaquetar en el nuevo contenedor, en lugar de transcodificarlas. Por último, las salidas también especifican un type de segmentado y un format de ts, más un url con una extensión de .m3u8, para que Zencoder sepa crear archivos HLS segmentados.

    Finalmente, también hay una lista de reproducción generada que hace referencia a cada una de las salidas segmentadas. Tenga en cuenta que el primer archivo al que se hace referencia en la lista de reproducción es el vídeo de baja velocidad de bits; el dispositivo que reproduce el vídeo generalmente cargará inicialmente la primera secuencia listada en la lista de reproducción y cambiará a otras secuencias según sea necesario. Esto significa que los primeros 10 segundos de reproducción utilizarán esta secuencia, después de lo cual el dispositivo tendrá suficiente información para seleccionar la secuencia apropiada para reproducir.

    NOTAS

    • La generación de listas de reproducción HLS actualmente no utiliza información de las otras salidas del trabajo, sino que es simplemente una forma de generar fácilmente una lista de reproducción de velocidad de bits adaptada con formato correcto y tenerla cargada junto con los otros archivos.
    • Mientras que los dispositivos iOS reproducirán la lista de reproducción de velocidad de bits adaptable creada, VLC no lo hará debido a errores en el manejo de URL relativas. Buscará los archivos de segmento en el mismo directorio que el archivo de lista de reproducción, en lugar del directorio del archivo de manifiesto.
